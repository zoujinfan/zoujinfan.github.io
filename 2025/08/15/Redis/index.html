Redis介绍

本人在运维项目中担任OA角色，运维过程中涉及Redis数据库知识。本文主要对书籍《Redis实战》进行大致了解，对OA需了解的知识点进行梳理整理，主要总结基本概念，实战部分咱不涉及。

一、初识Redis

  • Redis简单介绍

相对于普通数据库（mysql）数据存在硬盘，Redis是存在内存中。

-> 特点：访问速度快

-> 适用场景：存临时的、需快速访问的数据。例如：游戏排行榜、视频播放量（计数器）、临时验证码（会话存储）

  • Azure Cache for Redis

Azure云平台提供了Redis服务，称“Azure Cache for Redis”

图片解释：

1. 有缓存命中（Cache Hit） ✅

  • request → Azure Redis
  • Redis 里有数据 → 直接 response 给请求方

2. 缓存未命中（Cache Miss，数据库有数据） ✅

  • request → Azure Redis
  • Redis 没有数据 → 由你的应用去访问数据库
  • 数据库返回数据 → 你的应用把数据写回 Redis 做缓存，然后再返回给用户

注意：Redis 不会主动去查数据库，它只负责存和取。去数据库拿数据的逻辑是你代码里写的。

3. 缓存未命中（数据库也没数据） ✅

  • request → Azure Redis → Redis 没有
  • 应用去数据库 → 数据库也没有
  • 应用返回空给用户（通常不会往 Redis 里写空值，除非做“空缓存”防穿透）。

二、Redis特性（数据安全与性能保障）

复制

原文表达：

需要扩展读请求，或者在需要写入临时数据的时候，用户可以通过设置额外的Redis从服务器来保存数据集的副本。

在接收到主服务器发送的数据初始副本（initial copy of the data）之后，客户端每次向主服务器进行写入时，从服务器都会实时地得到更新。

在部署好主从服务器之后，客户端就可以向任意一个从服务器发送读请求了，而不必再像之前一样，总是把每个请求都发送给主服务器（客户端通常会随机地选择适用哪个从服务器，从而将负载平均分配到各个从服务器上）。

图中说明：

RDB 在 Redis 里是 Redis DataBase file 的缩写，指的是 快照持久化（Snapshotting） 生成的文件。

从服务器在进行同步时，会清空自己的所有数据

  • 主从复制（Master-Slave Replication）

    主服务器的数据（master） 完整拷贝到 从服务器（slave）上，并且保持同步。（第一次连接时，全量复制；后续再有更新，立马主同步到从上）

    优点：

    1. 读写分离：主机负责写，从机负责读

    2. 高可用：主机挂了，从机顶

    3. 数据备份：防止单机数据丢失

    • 主从链

      原文表达：随着负载不断上升，主服务器可能会无法快速地更新所有从服务器，或者因为重新连接或者重新同步从服务器而导致系统超载。为了缓解这个问题，用户可以创建一个由Redis主从节点（master/slave node）组成的中间层来分担主服务器的复制工作。

      选择主从链的主要场景：

持久化（将数据存储到硬盘）

  • RDB（Redis DataBase file, 快照snapshotting持久化生成的文件，某一时刻的所有数据写入硬盘），默认文件名dump.rdb

  • AOF（append-only file, 只追加文件，在执行写命令时将被执行的写命令复制到硬盘），Redis会把每条写命令记录在日志文件（appendonly.aof）里

三、处理系统故障

  1. 验证持久化文件是否完好无损（无论是快照持久化还是AOF持久化），并在有需时对其修复

  2. 更换故障主服务器

    方法 简单介绍
    Redis Sentinel 当主节点故障时，Sentinel 自动检测并将某个从节点提升为新的主节点（自动故障恢复）
    Redis Cluster Redis Cluster 中每个主节点都有对应的从节点，主节点宕机时从节点自动接管（自动分片和故障转移，需要客户端支持）
    手动切换 确认主节点故障，停止主节点服务，从节点提升主节点（小规模或临时应急）
    热备切换 准备好一台新的 Redis 服务器，将新的 Redis 配置为从节点，同步数据，等待数据同步完成后，将该节点提升为主节点（手动或自动），关闭旧主节点，业务端切换到新主节点（需要平滑切换的场景）

四、其他知识点

  1. 客户端分片（client-side sharding）:指由客户端负责将数据分布到多个 Redis 实例上，而不是由 Redis 服务器自身来管理数据分片。客户端根据一定的规则（如哈希算法）决定将某个 key 存储到哪个 Redis 实例。缺乏自动故障转移，某个 Redis 节点宕机，客户端需手动处理或重新映射分片。

  2. 键的过期时间：通过Redis的过期时间（expiration）特性来让一个键在给定的时限（timeout）之后自动被删除。

五、运维真实案例

问题：微软host os升级导致Redis（缓存服务器）故障转移（failover）

采取的措施：

  1. Redis CPU 高时采取重启策略，属于快速恢复服务可用性的“应急手段”，能短时间解除阻塞和异常状态。然后再重启业务服务，以重建与 Redis 的连接并刷新缓存，确保两者状态同步，保证业务稳定运行。

  2. 增加Redis服务副本（replica，也称“从节点”或“副本节点”），主要体现在高可用性、读性能提升，故障恢复能力增强等方面。

  3. HA（High Availabilty, 高可用性），是保障服务“持续在线”的核心能力，依靠主从复制、自动故障转移、负载均衡等机制，在故障时自动或快速恢复服务，避免影响用户体验。

其他说明：

Redis是运行在主机内存中的进程，当宿主机操作系统（Host Operation System）升级时，通常升级伴随内核升级、系统服务更新，很多升级操作必须重启才能生效，宿主机重启 redis进程就会停止。
